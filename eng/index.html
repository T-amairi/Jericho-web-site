JERICHO


New consensus


We want to see a decentralized, IoT-friendly cryptocurrency on the market. However, the crypto-currencies centered around the IoT are centralized, which is the opposite of what a crypto-currency should provide according to us. This is why we will during this project, propose a concept of crypto-currency adapted to the IoT market.
We want it to check scalability, security and low cost properties. For this, we have built our own consensus named Jericho.

Due to the decentralized nature of the network, consensus is an essential protocol that allows transactions to be conducted securely and prevents fraudulent transactions from being carried out.
Simply put, consensus is the way entities in the network agree on the validity of a transaction.
We have divided our network into nodes, and thus, during a transaction, the consensus will only take place in the lowest level nodes, directly concerned by this transaction. We have several architectures, based on different elements, either geographical or merit-based.

We will implement a mobile node module that will take care of randomly sending transactions to fixed nodes using an exponential law. Moreover, clusters will only be present at the leaf level. First, the mobile node generates a transaction and sends it to two leaves to be validated. The fixed nodes that received the transaction initiate a consensus within their cluster respectively. If they cannot initiate a consensus because it is not yet their round, the transaction is hidden in a queue to be processed later. When the consensus result is obtained, it is sent to the smallest common ancestor between the two fixed nodes processing the request. When routing the request to the responsible fixed node, the path is recorded with the transaction, i.e., which nodes forwarded the request. When the transaction is received by the responsible fixed node, it checks the result of the consensus of both fixed nodes. If one of them rejects the transaction, the transaction is also rejected. The response is then sent to both fixed nodes using the registered paths. As with the fixed nodes, a responsible fixed node can hide a request until it receives the result of the second to receive the result of the second consensus. Finally, the result is also broadcast throughout the network and the fixed nodes send the result of the transaction back to the mobile node.

NETWORK


Network Structure


To better understand the consensus invoked, we need to discuss the structure of the network in more detail. We will distinguish two types of nodes in the network, namely fixed and mobile nodes. Fixed nodes form clusters and have a single purpose: to validate transactions, update the local ledger and finally disseminate its new version. They cannot move, however, they have different roles to fulfill. They can be broadcasters or managers. As for the mobile nodes, they are simply the users of the network and the only action they can perform is to issue transactions to the fixed nodes. transactions to the fixed nodes. Moreover, because of their character, they can connect and disconnect at will from one cluster to another. These nodes represent, in other words, any IoT object, i.e.: phones, connected watches, computers, payment terminals, etc. payment terminals, etc.

Among the fixed nodes are the broadcast nodes that act as intermediaries between the network and the mobile nodes. They receive transaction requests from the nodes and then broadcast them to their fixed parent nodes. Finally, when a transaction is validated by the network, they communicate it to the mobile nodes concerned. There are also responsible nodes that have no communication with mobile nodes and are in contact only with fixed nodes. Their role is to validate transactions at their level, i.e., to verify in their local ledgers that the transactions are free of conflicts and to disseminate the transactions. They have the role of validating the transactions at their level, i.e. checking in their local ledgers if the transactions have no conflicts and broadcasting the validated transactions to the whole network so that the fixed and mobile nodes can update their local ledgers.

The structuring of the network is done via an n-ary tree where the leaves represent only fixed broadcasting nodes and the rest of the fixed responsible nodes. The important notion behind this structuring is that it allows a geographical decomposition by different levels: by city, by department, by region, by country and by continent. The number of levels has no limit and depends on the desired implementation.

DISCOVERY


Frequent questions


What is a ledger? It is a digital transaction recording system in which transactions and their details are recorded in one or more in one or more places at the same time.


What is the Internet of Things? The Internet of Things brings together the Internet and objects, places and physical environments. It is about objects connected to the Internet allowing a communication between physical goods and their digital and their digital existence, as a sensor.


Why create Jericho instead of using Proof of Work? Although Proof of Work is a consensus algorithm that masters the security aspect, its rigidity weakens its scalability because of the important time brought to the calculation. Moreover, Proof of Work can only work with transaction fees in order to remunerate the miners. It is therefore not adapted for the Internet of Things.
